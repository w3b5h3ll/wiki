Fuzzing101-3: TCPdump

发表于 2024-03-15 更新于 2024-04-07 阅读次数： Valine：

https://github.com/antonio-morales/Fuzzing101/tree/main/Exercise%203

In this exercise we will fuzz TCPdump packet analyzer. The goal is to find a crash/PoC for CVE-2017-13028 in TCPdump 4.9.2.

CVE-2017-13028: https://cwe.mitre.org/data/definitions/125.html

学习目标

  • What is ASan(Address Sanitizer), a runtime memory error detection tool

  • How to use ASAN to fuzz a target

  • How much easy is to triage the creashes using ASan

环境搭建

AFL++（注意使用llvm-15以及clang-15）: https://github.com/AFLplusplus/AFLplusplus/blob/stable/docs/INSTALL.md

  • 如无需FRIDA mode|QEMU mode|unicorn mode|...，直接make all即可

WSL性能加速

为了更快的性能，于WSL2.0重现了一套环境，如下

fuzz

语料库精简

afl-cmin，删除达到同样路径的冗余文件

afl-tmin默认只能单次执行一个输入，使用脚本循环进行处理

afl-system-config: 获得最佳fuzz性能

使用-M|-S指定多线程操作

此时，均只启用了havoc模式

Keep your test case small, Why?

在执行过程中，提示我们的某些case太大，有168KB

AFL的性能提升建议将此条作为了最重要的第一步，减小用例大小十分必要。其假设了一个场景:

如果翻转第47bit位置会导致bug，那么对于一个100字节大小的case，执行1000次发现此bug的概率，计算如下

$$
1-(1-\frac{1}{8*100})^{1000}\approx71%
$$

  • 随着case的size增大，发现bug需要执行的次数会显著增加

  • 运行时的内存，执行开销也会显著增大

此时，我们需要使用afl-cmin以及afl-tmin工具精简语料，并设置较好的超时时间-t（默认1000ms），限制其每次执行的时间。
